# buddyx-forge v2.0 — Feature Roadmap & Research

**Date:** April 7, 2026
**Status:** Planning reference (reviewed, issues fixed)
**Based on:** Deep research of Claude Code ecosystem, competing plugins, Anthropic documentation, community best practices

---

## Executive Summary

The Claude Code platform has evolved significantly since buddyx-forge v1.0 shipped. The ecosystem now has 340+ plugins, 1,300+ skills, native Agent Teams, an Agent SDK, and a new skills architecture that replaces the old commands pattern. buddyx-forge is still **the only plugin that generates a complete multi-agent system from scratch** — that's its moat. But the generated output uses patterns that are now legacy. This roadmap addresses that gap.

**Bottom line:** buddyx-forge's *generator* is excellent. The *output it generates* needs to catch up with the platform.

---

## Part 1: Platform Changes That Affect buddyx-forge

### 1.1 Skills Replaced Commands (CRITICAL)

Claude Code now treats `skills/` as the standard and `commands/` as legacy. From the official docs:

> "Files in either location create the same /slash-command interface. The skills system is the recommended approach going forward because it supports features that plain commands don't."

**What this means for buddyx-forge:** The plugin's `commands/` directory still works but misses these capabilities that only `skills/` supports:
- Progressive disclosure (subdirectories with references, scripts, examples)
- `context: fork` for isolated execution
- `!command` syntax for dynamic shell output injection
- Hooks in frontmatter
- `user-invocable: false` / `disable-model-invocation: true`
- `$ARGUMENTS` parameter passing

**Action:** Migrate all 11 commands to skills format. Keep `commands/` as fallback for backward compatibility.

### 1.2 Agent Teams Are Native Now

Claude Code v2.1.32+ supports native Agent Teams — multiple Claude instances coordinating through shared task lists and peer-to-peer messaging. This is a fundamentally different architecture than buddyx-forge's subagent-based team-lead.

| Feature | buddyx-forge team-lead | Native Agent Teams |
|---------|----------------------|-------------------|
| Coordination | Single subagent dispatches others | Shared task list + direct messaging |
| Context | All agents share parent's context | Each agent has isolated 1M context |
| Parallelism | Sequential or limited parallel | True parallel with worktree isolation |
| Communication | Report back to parent only | Peer-to-peer messaging between teammates |
| Token cost | Lower (shared context) | Higher (separate context per agent) |

**What this means:** buddyx-forge should support BOTH patterns:
- Subagent mode (current) for simple, cost-effective tasks
- Agent Teams mode for complex multi-domain work

### 1.3 New Hook Events

Claude Code now supports 21 hook event types with multiple handler types (command, HTTP, prompt, agent).

**New since v1.0:**
- `TeammateIdle` — fires when an agent team member is about to go idle
- `TaskCreated` — fires when a task is created in agent teams
- `TaskCompleted` — fires when a task is marked complete (quality gate!)
- `SessionStart` — fires at session start (context injection!)
- `PostToolUseFailure` — fires when a tool fails
- `Notification` — fires for notifications

**New hook handler types:**
- HTTP hooks (POST to a URL)
- Prompt hooks (single-turn LLM evaluation — uses Haiku, adds latency + cost per invocation)
- Agent hooks (spawn a subagent to verify conditions)

### 1.4 Progressive Disclosure Is Now Best Practice

Anthropic's official guidance: SKILL.md should be under 500 lines. buddyx-forge generates:
- Audit skill: **703 lines** (over limit)
- Diagram skill: **955 lines** (way over limit)

These should be split into SKILL.md (entry point) + reference files in subdirectories.

### 1.5 `!command` Dynamic Context

Skills can now embed shell commands that execute before the prompt reaches Claude:

```markdown
## Current Branch
!`git branch --show-current`

## Recent Changes
!`git log --oneline -5`
```

This replaces buddyx-forge's `inject-prompt-context.sh` hook with a cleaner, built-in mechanism.

### 1.6 `context: fork` for Isolated Execution

Skills can declare `context: fork` in frontmatter to run in a separate subagent context window. This is perfect for review and audit operations that shouldn't pollute the main context.

### 1.7 Built-in Skills Overlap

Claude Code now ships with built-in `/simplify`, `/review`, `/batch`, `/loop`, `/code-review`. Anthropic's own `/code-review` plugin uses 5 parallel Sonnet agents. buddyx-forge's review agent should differentiate by being **project-aware** (knows your domains, rules, conventions).

### 1.8 Native Token Monitoring

Claude Code has built-in `/context` and `/cost` commands that show token usage breakdown. No need to build a custom token monitor — integrate with existing tools instead.

## Part 2: Feature Roadmap

### Tier 1 — Architecture Modernization (v2.0)

**Realistic timeline: 3-4 weeks** (not 1-2 weeks — F1 alone requires generator + template + plugin restructure)

#### F1: Migrate Commands → Skills
**Impact:** Critical — fixes legacy pattern
**Effort:** Medium-High (touches generator, all templates, plugin structure)

Migrate all 11 commands from `commands/*.md` to `skills/*/SKILL.md`. Add progressive disclosure:

```
plugins/buddyx-forge/
  skills/
    setup/
      SKILL.md          # Entry point (<500 lines)
      references/        # Framework detection, config schema
      scripts/           # generate.py and builders
    scan/
      SKILL.md
    health/
      SKILL.md
    quick/
      SKILL.md
    ...
```

**Backward compatibility:** Keep `commands/` with thin wrapper files that reference the skill. Example:
```yaml
---
name: setup
description: See /buddyx-forge:setup
---
This command has moved to a skill. Use `/buddyx-forge:setup` directly.
```

**Generator changes needed:**
- New template structure for skills (SKILL.md + references/)
- Update `generate.py` to create skills/ directory instead of commands/
- Update plugin.json to reference skills instead of commands
- Update all internal cross-references (health check, scan, etc.)

**Tests needed:** Verify skills discovery, verify backward compat wrappers work.

#### F2: Split Oversized Skills
**Impact:** High — token efficiency
**Effort:** Low-Medium

Split the generated audit skill (703 lines) and diagram skill (955 lines):

```
skills/{project}/audit/
  SKILL.md              # Entry point (~200 lines)
  references/
    db-analysis.md      # Database table analysis
    code-quality.md     # Code quality checks
    security.md         # Security analysis
  templates/
    module-audit-template.html

skills/{project}/diagram/
  SKILL.md              # Entry point (~200 lines)
  references/
    er-diagram.md       # ER diagram generation
    flowchart.md        # Flowchart generation
    mermaid-patterns.md # Mermaid.js patterns
```

**Generator changes:** Split `audit-skill.tmpl` and `diagram-skill.tmpl` into entry point + reference templates.

#### F3: Add `context: fork` to Review & Audit
**Impact:** High — keeps main context clean
**Effort:** Low

Add `context: fork` frontmatter to the generated review agent and audit skill:

```yaml
---
name: {project}-review
context: fork
# ... rest of frontmatter
---
```

This means review/audit operations run in isolated context windows and only the summary comes back to the main session.

**Generator changes:** Add `context: fork` to `build_review_agent()` in `builders/agents.py` and audit skill template.

#### F4: Add New Hook Events to settings.json
**Impact:** Medium — leverages new platform capabilities
**Effort:** Low

> **Correction from original:** Hooks live in `settings.json` (or `settings.local.json`), NOT a separate `hooks.json` file. Claude Code reads hooks from settings files only.

Add support for new hook events in the generated `settings.json`:

```json
{
  "hooks": {
    "TaskCompleted": [{
      "matcher": "{project}-*",
      "hooks": [{"type": "command", "command": ".claude/scripts/eval/validate-task.sh"}]
    }],
    "SessionStart": [{
      "hooks": [{"type": "command", "command": ".claude/scripts/inject-session-context.sh"}]
    }]
  }
}
```

**Generator changes:** Update `build_settings_json()` in `builders/settings.py` to include TaskCompleted and SessionStart hooks when eval is enabled.

### Tier 2 — New Features (v2.1)

#### F5: Agent Teams Support
**Impact:** Very high — the biggest evolution in Claude Code
**Effort:** High

Add a new config option `"teamMode": "subagent" | "teams"` and generate accordingly.

> **Removed "auto" mode** — the decision logic was undefined. Users should explicitly choose based on their needs. Document the tradeoffs in setup.

When `teamMode: "teams"`:
- Team-aware CLAUDE.md with teammate creation instructions
- TaskCompleted hooks as quality gates (tests + review must pass)
- TeammateIdle hooks to prevent premature completion
- Worktree configuration for parallel work
- New `/buddyx-forge:team` skill to spawn pre-configured teams

**Decision criteria for users (documented in setup):**

| Use subagent when... | Use teams when... |
|---------------------|-------------------|
| 1-2 domains involved | 3+ domains involved |
| Cost-sensitive | Context-window-sensitive |
| Sequential work | Truly parallel work |
| Simple bug fixes | Multi-domain features |

#### F6: Dynamic Context with `!command`
**Impact:** High — eliminates inject-prompt-context hook
**Effort:** Low

Add `!command` syntax to generated SKILL.md files. Mark `inject-prompt-context.sh` as deprecated (keep for backward compat, document replacement).

#### F7: `/buddyx-forge:pr-review` Skill
**Impact:** High — directly competitive with Anthropic's /code-review
**Effort:** Medium

Generate a PR review skill that's **project-aware** (the differentiator):
- Dispatches domain-specific reviewers (knows which agent owns which files)
- Uses the project's RULES.md (framework-specific checklist)
- Checks against shared-learnings.md (learned patterns)
- Runs in `context: fork` (doesn't pollute main context)
- Outputs a structured report with pass/fail per domain

#### ~~F8: Token Budget Monitor~~ → F8: Context Tips in SessionStart
**Impact:** Medium
**Effort:** Low

> **Correction from original:** Claude Code already has `/context` and `/cost` commands built-in. Building a custom token monitor is redundant and unreliable.

**Instead:** Add a `SessionStart` hook that prints a quick tip:
```
buddyx-forge: Tip — Use /context to check token usage, /cost for spending.
```

And document in CLAUDE.md when to compact:
```
## Context Management
- Run `/context` if responses slow down
- Auto-compact triggers at ~83.5% usage
- PreCompact hook preserves: task goal, files changed, agent delegations
```

#### F9: Sandbox Configuration
**Impact:** Medium — reduces permission fatigue
**Effort:** Low

> **Correction from original:** Sandbox uses a `sandbox` key in settings with `writableDirectories` and `allowedDomains` arrays, not the `fileAccess`/`networkAccess` format originally shown.

Generate sandbox settings in `settings.json`:

```json
{
  "sandbox": {
    "writableDirectories": [".claude/", "src/", "app/", "tests/"],
    "allowedDomains": ["localhost"]
  }
}
```

**Generator changes:** Add framework-aware writable directories (e.g., `vendor/` for Laravel, `node_modules/` for JS).

### Tier 3 — Growth Features (v2.2+)

> **Trimmed from 6 to 3 features.** F10 (SDK), F11 (monorepo), and F12 (mobile) are speculative — no users have requested them. Add them back when there's demand.

#### F13: Test Generation Skill
**Impact:** High for adoption
**Effort:** Medium

Generate a `/buddyx-forge:test` skill that creates test files:
- Framework-aware (PHPUnit, Jest, pytest, go test, RSpec)
- Reads the domain's model/service files
- Generates test skeletons with edge cases
- Uses `context: fork` to not pollute main context

#### F14: Hookify Integration
**Impact:** Medium
**Effort:** Low

Direct integration with Anthropic's hookify plugin:
- Generate rules in hookify's format
- Add `/buddyx-forge:hookify` that converts rules to active hooks

#### F15: Prompt Hooks for Smart Validation
**Impact:** High — leverages new Claude Code capability
**Effort:** Medium

> **Cost warning:** Each prompt hook invocation runs a Haiku inference. On a project with 100 tool calls/session, that's 100 extra API calls (~$0.10-0.50). Only use for high-value checks, not every tool call.

Use sparingly for domain ownership enforcement:

```json
{
  "type": "prompt",
  "prompt": "The agent is about to modify {file}. Based on the domain map, does this file belong to the active agent's domain? Answer YES or NO.",
  "model": "haiku"
}
```

**Recommendation:** Make this opt-in (`"promptHooks": true` in config) and only attach to Write/Edit, not all tool calls.

## Part 3: Competitive Landscape

### What buddyx-forge does that nobody else does:
1. **Generates a complete multi-agent system** from 10 questions
2. **Framework-aware** — 7 frameworks with specific rules, permissions, scan paths
3. **Self-improving memory** — shared learnings across agent sessions
4. **Domain-based routing** — keyword-to-agent mapping

### What competitors do that buddyx-forge should adopt:

| Feature | Who Has It | Priority |
|---------|-----------|----------|
| Progressive disclosure | Everyone (best practice) | **Must-have** |
| Agent Teams support | OMC, Harness | **Must-have** |
| PR review workflow | Anthropic /code-review | **Should-have** |
| Dynamic `!command` context | Many skills | **Should-have** |
| context: fork for review | Best practice | **Should-have** |
| Sandbox configuration | Best practice | **Should-have** |
| Test generation | Demand-driven | Nice-to-have |

## Part 5: Breaking Changes in v2.0

1. **SKILL.md body limit** — Generated skills must stay under 500 lines
2. **commands/ → skills/** — Plugin structure changes (backward compat wrappers maintained)
3. **New hook events** — TaskCompleted, SessionStart added to settings.json
4. **context: fork** — Review agent runs in isolation (changes behavior — summary only returns to main context)
5. **sharedDb path** — `../` paths now rejected (security fix from v1.1.1)
6. **Framework validation** — Unsupported frameworks now error (was silent fallback to Laravel)

## Part 6: Migration Path (v1.x → v2.0)

> **This was missing from the original roadmap.**

The `/buddyx-forge:upgrade` command needs to handle:

1. **Detect version** — Read `buddyxForgeVersion` from settings.json
2. **Backup** — `cp -r .claude/ .claude.backup.{timestamp}/`
3. **Preserve customizations** — Extract custom constraints, manual domain-map entries, custom CLAUDE.md sections
4. **Regenerate** — Run generator with detected/confirmed config
5. **Restore customizations** — Re-insert preserved content
6. **Verify** — Run `/buddyx-forge:health` automatically

The upgrade command already exists (`commands/upgrade.md`) but needs testing against the v2.0 output structure.
